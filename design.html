PPTX → PptxGenJS
代码转换器

一个 Node.js 工具，输入 .pptx 文件路径，输出一份可直接运行的 PptxGenJS 生成脚本及媒体资源文件夹。以 npm 包形式发布，同时提供编程 API 与 CLI 两种使用方式。

运行环境Node.js 发布形态npm 包 + GitHub 开源 输出格式PptxGenJS JS 脚本 还原目标可接受且可运行
§ 01 项目定位与核心哲学
"PptxGenJS 能力边界即转换边界"
能做的

精确转。PptxGenJS API 能原生支持的一比一精确映射：文本框、图片、基本形状、纯色填充、表格、图表。

做不好的

退化转。渐变→纯色、复杂动画→淡入、SmartArt→截图（或提取文本列表）作为 fallback。

做不了的

静默跳过。ActiveX、OLE 嵌入对象、3D 模型、VBA 宏、嵌入字体直接丢弃，日志记录但不中断流程。

不做什么

  • ✗ 不做双向转换（js 回 .pptx）
  • ✗ 不做在线平台或图形界面
  • ✗ 不对 PptxGenJS 做扩展或补丁
  • ✗ 不追求 100% 像素级还原度

发布形态

工具以 npm 包形式发布，同时托管在 GitHub 作为开源项目。两种使用方式共用同一套核心逻辑，互不干扰：

CLI 使用方式
npx pptx2js input.pptx -o ./out

面向非集成场景：一次性文件转换、本地批处理、快速验证。无需写代码，安装即用。

编程 API 使用方式
const { convert } = require('pptx2js');
await convert(filePath, options);

面向集成场景：CI/CD 流程、Electron 应用、自动化脚本，可编程控制转换选项和结果处理。

package.json 同时声明 main（库入口 lib/index.js）和 bin（CLI 入口 bin/pptx2js.js），CLI 层只是对核心 API 的薄封装，不含独立逻辑。

§ 03 系统架构

整个工具由六个独立模块组成流水线，各模块除主链强顺序依赖外，其余可并行设计。

📄 input.pptx
① 解压与索引
JSZip 解压，建立文件清单；解析 [Content_Types].xml 和 _rels/*.rels 构建关系 ID 映射
② XML 预解析
xml2js 批量解析关键 XML（统一配置），构建内存对象树；仅解析转换所需文件
③ 实体提取器
遍历幻灯片 spTree，能力检查矩阵标记 FULL / DEGRADE / SKIP；完成三层属性合并
④ 映射引擎
翻译为平台无关 IR；完成 EMU→英寸转换、颜色规范化（含主题色解析）、Z-order 排序
⑤ 代码生成器
IR 序列化为格式化 JS；自实现缩进层级拼接（不依赖 Prettier）
⑥ 资源打包器
复制 media/ 文件；处理重名冲突；写入退化生成图片
📄 output.js 🖼 media/ 📋 conversion.log
§ 04 关键难点与设计决策
4.1 母版与版式的继承链

PPTX 继承链：slideMaster → slideLayout → slide。一个文本框可能只有少数属性写在 slide 层，其余继承自 layout 和 master 层。

属性解析在实体提取器阶段完成，合并后的实体带完整属性进入映射引擎。

⚠ 实现注意 — 属性合并是最易出 bug 的环节

多个占位符 idx 对应、版式层缺失对应 sp 等情况极易引发静默错误。须明确定义属性合并优先级，并在代码中显式处理每一层缺失的情况：

属性合并优先级（高 → 低）
slide 层 <p:sp>
›
slideLayout 层对应 <p:sp>
›
slideMaster 层对应 <p:sp>
›
PptxGenJS 默认值

匹配依据：占位符 <p:ph idx="N"> 相同的 idx 值。若 layout 层无匹配 sp，直接向上追溯至 master 层，不报错。

4.2 SmartArt 的退化处理

SmartArt 在 PPTX 中是 <dgm:relIds> 引用的 Diagram 数据，PptxGenJS 完全无法处理。

⚠ 实现注意 — 次选策略须提取文本而非留空

若次选直接生成空占位矩形，用户将完全丢失 SmartArt 中的信息。应从 dgm:data 提取 <a:t> 文本节点，退化为带编号的文本列表，保留内容可读性。

  1. 优先：检查 slideN.xml.rels 中是否有与 SmartArt 关联的 .png 缓存图片（PowerPoint 原生生成），有则作为图片插入，尺寸与原 SmartArt 一致。
  2. 次选：无缓存图片时，从 dgm:data XML 中提取所有 <a:t> 文本节点，生成带编号的 addText() 列表，占位原始区域。同时在日志中记录原始节点结构摘要。
  3. 兜底：若 dgm:data 也无法读取，生成占位矩形 + "[SmartArt 无法解析]" 文本，并在转换日志中标注 severity: error。
4.3 图表的处理

PptxGenJS 的 addChart() 支持约 10 种类型（BAR、LINE、PIE、SCATTER 等），无法覆盖 PowerPoint 20+ 种图表（瀑布图、树状图、漏斗图等）。

对不支持的类型：检查是否有缓存图片，有则图片插入；无则占位文本 + 日志警告。对支持的类型，从 <c:chart> XML 提取分类轴数据（<c:cat>）、数据系列（<c:ser>）、图表标题（<c:chartTitle>）。

4.4 组合形状的展平

PPTX 的 <p:grpSp> 拥有自己的坐标系（子形状坐标相对于组合原点），且支持嵌套。

策略：递归展平。将子元素的相对坐标加上组合的 <a:off> 偏移，转换为绝对坐标后作为独立形状输出。嵌套组合递归处理，展平后按原始出现顺序排列以保留 Z-order。

4.5 表格的映射

PptxGenJS 的 addTable() 支持度较高，映射相对直接。

  • · 从 <a:tblGrid> 提取列宽
  • · 从 <a:tr> 逐行提取 <a:tc> 单元格，每格内 <a:r> 独立格式化
  • · 合并单元格：解析 rowSpan / gridSpan 属性，映射到 PptxGenJS 的 merge 格式
  • · 单元格边框：从 <a:tcPr> 的 lnL / lnR / lnT / lnB 提取
4.6 xml2js 统一配置规范
⚠ 实现注意 — 全项目必须使用统一解析配置

xml2js 对命名空间前缀（如 a:r vs r）的处理极为脆弱，不同模块若使用不同配置会导致解析结果不一致。全项目必须使用以下统一配置，通过单一入口文件导出，禁止在模块内自行实例化解析器：

// xml-parser.js — 项目唯一导出的解析器实例
const xml2js = require('xml2js');

const PARSER_OPTIONS = {
  explicitArray: false,     // 单子节点不包裹数组
  mergeAttrs: false,       // 属性保留在 $ 键下，不合并
  explicitCharkey: false,  // 纯文本节点直接为字符串
  tagNameProcessors: [],   // 保留原始命名空间前缀
  attrNameProcessors: [],  // 属性名不做处理
  xmlns: false              // 不展开命名空间 URI
};

module.exports = {
  parseXml: (str) => xml2js.parseStringPromise(str, PARSER_OPTIONS)
};

所有模块必须通过此统一入口解析 XML，禁止在模块内自行实例化解析器。访问节点时统一使用带命名空间前缀的键名，如 node['a:r']。

conversion.log 结构

JSON 格式，包含以下字段：

设计说明 — 补充机器可读字段以支持 CI 集成

每个退化/跳过条目新增 slideIndex、elementBounds（位置信息）、severity（warn / error）三个字段，方便外部脚本或 CI 流程处理转换结果。

字段说明
source源文件信息（路径、大小、幻灯片数）
statistics汇总：总幻灯片数、精确转换数、退化数、跳过数
slides[]按幻灯片组织的详细清单，每项含 slideIndex、elementBounds、decision、severity
degraded[]退化项列表，含原始结构摘要、退化方式、severity
omitted[]跳过项列表，含元素类型、跳过原因、severity
warnings[]全局警告（缺失字体、无法解析的结构等）
§ 06 CLI 接口设计
$pptx2js input.pptx -o ./output-dir [options]
设计说明 — --strict 按严重性拆分，避免误触

单一 --strict 无法区分"空占位符被跳过"和"核心图表无法转换"这两种完全不同的严重性。拆分为 --strict-degrade 和 --strict-skip 两个独立选项，分别控制。

参数默认值说明
input.pptx必填源 PPTX 文件路径
-o, --output./pptx2js-output输出目录
--no-mediafalse不提取媒体文件，图片引用变占位注释
--strict-degradefalse任意退化项触发非零退出码
--strict-skipfalseseverity:error 级别的跳过项触发非零退出码
--log-levelinfo日志详细程度：minimal / info / verbose
--max-file-size50MB超过此阈值时切换为逐幻灯片流式解析

§ 08 测试策略
8.1  单元测试 Jest
每个模块独立测试，输入为标准 XML 片段（直接用字符串），验证 IR 输出正确性。重点覆盖：属性继承合并、EMU 坐标换算、颜色主题解析、SmartArt 三层 fallback 分支。
8.2  集成测试：往返验证 最重要
golden.js → golden.pptx → pptx2js 转换 → generated.js → roundtrip.pptx

解压对比 golden.pptx 和 roundtrip.pptx 中 slide1.xml 的文本/位置/颜色是否一致，允许浮点舍入误差（±0.001 英寸）。

8.3  退化/跳过覆盖测试
专门测试 PPTX 文件包含 SmartArt、复杂动画、3D 模型等，验证工具不崩溃、日志 severity 字段正确、退化输出合理。
8.4  真实语料库回归测试
设计说明 — 手工金标准无法覆盖真实边界情况

手工制作的金标准只能测到预想到的情况，真实 PPT 的复杂组合无法预判，需要真实语料库作为补充。

收集 5–10 份真实业务 PPT（去除敏感信息），加入回归测试集。每次 PR 合并后自动运行转换，CI 检查 conversion.log 中 severity:error 数量是否超出阈值；关键幻灯片由人工目视对比（截图 diff）。

§ 09 已知局限与显式声明
  • L1 字体完全依赖宿主环境。代码中会写 fontFace: "思源黑体"，但若运行环境未安装该字体，PowerPoint 会回退到默认字体。这是 PptxGenJS 本身的局限，工具不负责字体检测或打包。
  • L2 动画还原度低。所有动画退化为淡入 0.5 秒，这是设计取舍而非缺陷。
  • L3 不保证二进制一致性。往返生成的 PPTX 与原始 PPTX 逐字节不同（XML 空白、命名空间前缀、元素顺序不同），但视觉呈现应一致。
  • L4 不处理受密码保护的 PPTX。加密文件直接报错退出，提示用户先解密。
  • L5 不处理 .ppt 格式（97-2003 旧格式）。仅支持 .pptx（OOXML）。
  • L6 不支持增量转换。每次转换都是全量重写，不支持"保留已有 JS 代码，只转换新增幻灯片"。
  • L7 大文件内存限制。全量 XML 解析对 200MB+ 大型 PPT 可能造成内存压力。超过 50MB 时自动切换为逐幻灯片流式解析（详见 CLI 参数 --max-file-size）。
